## 배경

`docs/superpowers/plans/2026-07-10-ai-server.md` Task 3(`gemini_client.py`,
`call_gemini_analyze`)는 이미 세션당 Gemini `flash-lite` 구조화 생성 호출
1회 안에서 분류(`type`: hook/claude_md)+커맨드·문구 생성+이유(`reason`)+
confidence를 모두 처리하도록 구현·테스트(4개 테스트 통과)되어 있었다.
`DESIGN.md` "프롬프트 설계 원칙" 절은 "한 번에 시키는 일의 개수를
최소화"할 것을 원칙으로 제시하면서 "분류+이유 설명+커맨드 생성을 억지로
다 우겨넣지 말고"를 예시로 들었는데, 이 원칙이 구체적으로 호출을
나누라는 뜻인지, 한 호출 안에서 스키마/few-shot으로 범위만 좁히면
되는 뜻인지는 명시돼 있지 않았다(`/plan-eng-review`에서 확정하기로
미뤄뒀던 항목, DESIGN.md 섹션5 TODO). Task 3는 이 해석을 검증 없이
"1콜, 스키마에 다 담기" 쪽으로 이미 선택한 상태였다.

한편 `DESIGN.md` "점진적 매칭" 절(2026-07-10 확정)은 별도로 "세션 업로드
→ 전처리(규칙 기반, AI 아님) → Gemini 구조화 생성 호출 1회 → 후보 추출"을
아키텍처로 이미 확정해뒀다. 이 화살표 체인에서 실제 Gemini API 호출은
"구조화 생성 호출" 한 단계뿐이고("전처리"는 규칙 기반 코드, "후보 추출"은
그 호출의 응답을 파싱하는 것) — 즉 세션당 호출 1회라는 아키텍처는
이미 확정돼 있었고, Task 3는 이를 만족한다.

## 결정

**Option A 채택 — 현행 1콜 설계를 그대로 유지한다.** 코드 변경 없음:
이미 구현된 Task 3(`call_gemini_analyze`)를 검증 근거와 함께 승인한다.

## 근거

1. `DESIGN.md` "점진적 매칭" 절이 이미 "세션당 구조화 생성 호출 1회"를
   아키텍처로 확정해뒀다 — Task 3가 이를 그대로 만족한다.
2. RPM 예산이 아직 실측 전이다(T-02 대기 중, 현재는 8~10 추정치). 호출
   수를 늘리는 결정을 실측 없이 지금 내릴 근거가 없다.
3. "flash-lite가 요구사항이 많으면 품질이 떨어진다"는 관찰은 아직
   가설이다. 실측(T-08) 없이 설계를 바꾸는 것은 이 프로젝트가 이미
   여러 번 써온 "실측 후 결정" 원칙(RPM 임계값 3명→2명 조정, 비동기
   큐 대신 `aiolimiter`로 대체 등)에 어긋난다.
4. 이미 구현·테스트된 것을 유지하는 게 가장 저비용이다 (Simplicity
   First / YAGNI).

## 폴백 기준 (T-08 실측 후 전환 조건)

T-08(로컬 실행 확인 + 품질 실측)에서 실제 curl 응답의 hook/claude_md
분류 정확도, reason 문장 품질을 확인했을 때 문제가 발견되면:

1. **먼저 Option C**(프롬프트/스키마 경량화)를 시도한다 — RPM 비용이
   없고 구현 비용이 작다.
2. C로도 부족하면 **Option B**(호출 분리)로 전환한다 — RPM 예산이
   실측된 뒤(T-02) 여유가 확인된 경우에만.

이 순서는 RPM 예산을 아끼는 저비용 대안부터 시도한다는 원칙을 따른다.
품질 문제가 없는 것으로 확인되면 Option A를 계속 유지한다.
