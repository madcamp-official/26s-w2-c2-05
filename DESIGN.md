# Design: Claude Code 세션 로그 → 팀 개발 환경 추천 툴

Generated by office-hours 브레인스토밍, 2026-07-10
Mode: 해커톤/데모 (몰입캠프 Week 2)
Status: DRAFT

## Problem Statement

비개발자 또는 그룹 프로젝트 팀원들은 Claude Code의 hook/CLAUDE.md/skill 같은
엔지니어링 컨벤션을 잘 모른다. 그 결과:
- AI를 비효율적으로 써서 토큰을 낭비하거나
- 팀원마다 제각각의 방식으로 AI를 써서 결과물 스타일이 통일되지 않는다.

## What Makes This Cool

세션 로그만 올리면 그 사람(또는 팀)에게 맞춘 hook/CLAUDE.md가 **자동으로 생성**된다는
것. 특히 팀 버전에서는, 여러 명이 각자 세션을 올리면 "이 팀은 이렇게 일하는구나"를
발견해서 팀 전체가 공유할 컨벤션을 뽑아준다 — 개인용 세션 분석기는 이미 여럿 있지만,
"여러 명의 세션을 병합해서 팀 공통 환경을 만들어주는" 도구는 없다.

## Premises

1. 타겟은 확정 — 엔지니어링 컨벤션을 모르는 비개발자/팀원. 번역 UX가 핵심 가치.
2. 차별화 포인트는 개인 추천이 아니라 **여러 명의 세션을 모아 팀 단위로 통일된
   개발 환경을 구축**하는 것.
3. 포트폴리오 어필 포인트: "우리가 Agent AI 워크플로우를 이해하고 있다"를
   보여주는 결과물이기도 함.
4. 서브 스토리: 토큰 낭비 방지 (초보자의 비효율적인 AI 사용 패턴을 잡아줌).

## Approaches Considered

### Approach A: 개인 코치 + 팀은 데모용 목업
개인 세션 업로드→추천 플로우만 탄탄히 만들고, 팀 화면은 준비된 데이터로 보여주기만.
공수 작음, 리스크 낮음. 단점: 차별화가 얕아 보임 — 심사위원이 파고들면 목업인 게 티남.

### Approach B: 진짜 팀 환경 빌더 (채택)
같은 프로젝트에 속한 여러 명이 각자 JSONL을 업로드 → 유저별 패턴 추출 →
"몇 명에게서 몇 번 반복됐는지" 기준으로 실제 병합 → 팀 공통 hook/CLAUDE.md 추천.
공수 중간~큼, 리스크 중간(다중 세션 테스트 데이터 확보, 병합 임계값 튜닝).
장점: 기존 도구에 없는 각도, 실제 팀 페인포인트(온보딩/컨벤션 통일)를 건드림.

### Approach C: 개인용 + 즉시 적용 UX 극한 추구
팀 기능은 로드맵으로 미루고 "적용" 버튼으로 실제 파일이 생성되는 경험에 집중.
공수 중간, 리스크 낮음. 단점: 이미 나온 가장 뾰족한 차별화 스토리를 포기하는 셈.

## Recommended Approach: B — 팀 환경 빌더

### 병합(merge) 로직 — 생각보다 크지 않다

기존 스키마의 `scope`/`affected_members` 필드에 실제 값을 채우는 문제로 축소된다.

1. **1단계 (개인 추출)**: 원래 설계대로 유저별 JSONL을 각각 전처리 → Gemini로
   `AnalysisResult` 생성. 이건 팀 모드에서도 동일하게 재사용.
2. **2단계 (그룹핑 — /plan-ceo-review에서 확정, 2026-07-10)**: 생성(글쓰기·판단)은
   LLM, 그룹핑·카운팅(정확해야 하는 작업)은 결정론적 코드+임베딩으로 역할을 분리한다.
   - Hook 후보: `event` + `matcher` + 정규화한 `command` 문자열이 동일하면 같은 그룹
     — 셸 명령어는 형태가 고정적이라 문자열 정규화만으로 충분, 임베딩 불필요.
   - CLAUDE.md 후보: `suggested_text`를 Gemini 임베딩 모델(`gemini-embedding-001`/`-2`,
     `task: clustering`)로 벡터화 → 코사인 유사도 기준으로 로컬에서 그룹핑
     (라이브러리 없이 벡터 내적만으로 충분, 팀당 후보 수십 개 수준이라 계산량 미미).
     임베딩 모델은 생성 모델과 별도 쿼터라 RPM 부담을 늘리지 않는다.
   - **병합용 재작성 LLM 호출은 없음.** 그룹의 대표 문구는 그룹 내에서 가장 신뢰도
     높았던 원문을 그대로 노출하고, 병합된 다른 표현들도 함께 보여준다
     ("3명이 이렇게 말했어요: ..." + 원문 목록) — 투명성과 정확한 카운팅을 동시에 확보.
3. **3단계 (팀 신뢰도 계산)**: 그룹 안에 몇 명의 서로 다른 유저가 있는지 코드로
   정확히 세서 `affected_members`에 채운다 (LLM에게 세게 하지 않음). 개인 모드의
   "3회 미만이면 추천 안 함" 규칙과 같은 원리로, "N명 미만이면 팀 추천에서 제외"
   임계값을 하나 정한다 (기본: 2명 이상).
4. **충돌 처리**: 정교한 다수결/최신우선 로직은 이번 캠프 기간엔 넣지 않는다.
   단순히 "가장 많은 사람이 반복한 패턴 순"으로 정렬해서 보여주는 것으로 충분 —
   이건 원래 스펙의 "로드맵" 항목이었지만, 최소 버전(카운트 기반 정렬)은
   1주일 안에 구현 가능한 범위다.

### Gemini Flash 무료 티어 제약 (2026-07-10 확인)

공식 문서(ai.google.dev)는 계정별 실시간 한도를 AI Studio 대시보드에서만 보여주고
고정 표를 공개하지 않는다. 서드파티 소스들은 `gemini-2.5-flash` 기준 10~15 RPM,
250~1500 RPD 사이로 엇갈리게 보고한다 — **정확한 값은 팀 계정의 AI Studio
대시보드에서 직접 확인할 것.** 보수적으로 10 RPM으로 잡아도: 팀 5명 기준 유저별
분석 호출 5회는 1분 안에 여유 있게 처리 가능 — 호출 "횟수"는 병목이 아니다.
진짜 비용 레버는 프롬프트 크기(이미 전처리 단계에서 원본 로그 대신 압축된 패턴
요약만 보내도록 설계되어 있어 해결됨)와 "병합 단계에 생성 호출을 추가하지 않는 것"
(위 2단계 참고 — 임베딩으로 대체해 생성 모델 RPM 예산을 아예 건드리지 않음).

### 왜 A/C보다 나은가

A는 원래 스펙에서 이미 "로드맵"으로 분류했던 걸 데모용 무대장치로만 쓰는 셈이라
얕아 보일 위험이 있다. C는 좋은 UX지만 이번 대화에서 나온 가장 뾰족한 차별화
스토리(팀 통일 환경)를 스스로 포기하는 선택이다.

## Success Criteria

- 3명 이상의 (실제 또는 합성) 세션 로그를 업로드했을 때, 팀 공통 추천이
  "몇 명 중 몇 명에게서 나왔는지" 근거와 함께 표시된다.
- 개인 모드와 팀 모드가 같은 백엔드 파이프라인(전처리 → Gemini 구조화 분석)을
  공유하고, 팀 모드는 그 위에 병합 단계만 추가된 구조다.
- 비개발자가 결과 화면만 보고 무슨 뜻인지 이해할 수 있다 (hook/CLAUDE.md 용어를
  풀어서 설명).

## Next Steps (1주 기준 대략적 순서)

1. 개인 모드 파이프라인 먼저 완성 (전처리 → Gemini 구조화 분석 → 추천 표시) —
   이게 팀 모드의 1단계이기도 하므로 선행 투자가 낭비되지 않는다.
2. 팀 모드 데모용 다중 세션 테스트 데이터 준비 (실제 팀원 로그가 없다면 합성).
3. 병합 로직(그룹핑 + affected_members 카운트) 구현.
4. 팀 결과 화면 — "N명 중 M명이 이 패턴을 반복했습니다" 같은 근거 표시.
5. (시간 남으면) Approach C의 "즉시 적용" UX를 개인 모드에 추가.
