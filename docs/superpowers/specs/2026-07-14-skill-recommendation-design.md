# Design: Skill 추천 타입 (hook/claude_md에 이은 3번째 타입)

Generated by superpowers:brainstorming, 2026-07-14
Status: DRAFT (승인 대기)

## 배경 — 왜 이 문서가 필요한가

`DESIGN.md` 섹션1은 원래 "hook/CLAUDE.md/**skill**" 세 가지 추천 타입을 그리고
있었지만, `SkillRecommendation` 스키마 초안만 남기고 실제 구현은 계속 미뤄왔다
(`DESIGN.md` "Next Steps" 4번: "hook/claude_md 두 타입이 안정화된 뒤에" — 이유는
skill이 hook/claude_md 양쪽 경계를 다 침범할 수 있는 가장 애매한 분류라, 안정된
2분류 베이스라인 위에 나중에 얹어야 실패 지점이 좁혀져 디버깅이 쉽기 때문).

이제 hook/claude_md 두 타입이 안정화됐고(AI서버/웹서버 연동 스프린트, 온보딩
스프린트, hooks 분리 스프린트까지 전부 완료·테스트 통과 확인됨), skill 타입을
얹을 차례다.

`DESIGN.md`의 "TODO(설계 미정 사항)" 섹션1/섹션3에 남아있던 미해결 항목 3개
(전처리 방법, 최소 반복 임계값, 팀 모드 병합 기준)를 이 문서에서 확정한다.
추가로, 브레인스토밍 중 스코프 자체가 이전 예상보다 커진다는 게 드러났다 —
"적용하기"가 실제로 무엇을 만드는지(SKILL.md 파일 생성/저장/PUSH까지)가
DESIGN.md에 전혀 정의돼 있지 않았기 때문.

## 확정된 설계 결정

| 미정 사항 (DESIGN.md TODO) | 결정 | 이유 |
|---|---|---|
| 전처리가 다단계 시퀀스를 어떻게 추출할지 | Bash 커맨드만 대상(v1), tool_use 확장은 v2 | hook과 동일하게 객관적 신호(정확한 커맨드 문자열)만 다뤄서 범위 최소화. DESIGN.md 예시("마이그레이션→시드→재시작")도 전부 bash 커맨드 |
| 시퀀스 탐지 방식 | 갭 허용(중간에 다른 명령이 껴도 같은 시퀀스로 인정) | 실제 워크플로우는 A 다음에 바로 B가 오는 경우가 드물고, 중간에 파일 편집 등 다른 행동이 섞이는 게 정상. 연속(contiguous) 매칭만 하면 대부분의 실제 패턴을 놓침 |
| 시퀀스 길이 | 최대 4단계까지 점진적 확장(Apriori 방식) | 2단계 쌍(A→B)부터 임계값 통과하는 것을 찾고, 통과한 쌍을 3단계로 확장 시도 → 계속 임계값 넘으면 채택(짧은 하위 체인은 버림). DESIGN.md의 3단계 예시를 실제로 잡아내려면 최소 이 정도 확장이 필요 |
| skill 후보 최소 반복 임계값 | 2회 (hook/claude_md는 3회) | 단계 수가 늘어날수록 정확히 3번 반복되기 어려움. 대신 노이즈(우연히 이어진 무관한 명령) 오인식 위험이 hook/claude_md보다 커진다는 트레이드오프는 감안 |
| 팀 모드 병합 기준 | Gemini가 생성한 `skill_description` 임베딩 코사인 유사도(claude_md와 동일 방식·임계값 0.85) | 사용자 선택 — 단계 순서/커맨드가 팀원마다 조금 달라도 "같은 의도"의 워크플로우면 하나로 묶이길 원함. (대안이었던 "정규화된 스텝 튜플 완전일치"는 기각 — hook만큼 엄격하게 매칭하면 팀원 간 사소한 변형에도 별도 그룹으로 쪼개질 위험) |
| "적용하기"의 실제 동작 범위 | 실제 `SKILL.md` 생성 + 저장 + PUSH까지 전부 구현 | 사용자 선택 — hook/claude_md처럼 카드만 보여주고 끝나면 skill 타입을 도입하는 의미(실제 Claude Skill 파일을 만들어주는 것)가 없다고 판단 |
| 스킬 생성 경로 | AI 후보 "적용하기"로만 생성 (v1은 수동 "새 스킬 만들기" 없음) | content/hooks는 처음부터 DEFAULT 값이 있어 수동 편집이 자연스럽지만, skill은 "없다가 생기는" 개념이라 v1은 AI 적용 경로만 지원해 UI/API 범위를 줄임. 생성 후 편집/삭제는 지원 |

## 아키텍처 개요

```
[웹서버] 세션 업로드 (기존 파이프라인, 변경 없음)
   ↓
[web-server/preprocessing.py] extract_pattern_summary 확장
   - 기존: 단일 bash 커맨드 반복 / 유저 정정 반복
   - 신규: 갭 허용 부분수열로 반복되는 커맨드 시퀀스(2~4단계) 탐지
   ↓
[web-server/ai_client.py] ai_client.analyze() (기존, 변경 없음 — pattern_summary 텍스트만 늘어남)
   ↓
[ai_server] /analyze — GeminiAnalyzeSchema에 skill_candidates 리스트 추가
   SYSTEM_INSTRUCTION에 3분류 기준(hook/claude_md/skill) 추가
   ↓
[web-server/matching.py] match_skill_candidate() 신규
   - claude_md와 동일: ai_client.embed(skill_description) → 코사인 유사도 ≥0.85
   - RecommendationGroup(type="skill") 생성/합류, GroupMembership 근거 기록
   - 2명 이상이면 promoted=True (기존 PROMOTION_THRESHOLD 그대로)
   ↓
[프론트] 개인 추천 / 팀 추천 카드에 skill 타입 렌더링 추가
   "적용하기" 클릭
   ↓
[웹서버] POST .../personal-recommendations/{id}/apply 또는
         .../recommendation-groups/{id}/apply (기존 엔드포인트, type 분기 추가)
   → type=="skill"이면 새 Skill row 즉시 생성(payload의 skill_name/
     skill_description/suggested_steps 사용)
   ↓
[프론트] 새로 생성된 Skill을 "Skill" 탭에서 바로 편집 가능 (이름/설명/steps)
   "저장" → PUT /projects/{id}/skills/{skill_id} → ProjectRevision(target="skill") 생성
   ↓
owner가 "PUSH" 클릭
   ↓
[웹서버] push_to_github — 기존 CLAUDE.md + settings.json push에 이어
   프로젝트의 모든 Skill을 .claude/skills/<name>/SKILL.md로 각각 push
```

## 컴포넌트 상세

### 1. 전처리 — 다단계 시퀀스 추출 (`web-server/preprocessing.py`)

기존 `extract_pattern_summary`는 단일 이벤트 반복(bash 커맨드 하나, 유저 정정
발언 하나)만 `Counter`로 세는 구조다. 여기에 시퀀스 탐지를 추가한다.

**입력**: 세션 안에서 `_extract_bash_commands`로 뽑은, 실행 순서 그대로의
정규화된 커맨드 리스트 (기존 `Counter` 집계 이전 단계의 원본 순서 리스트가
필요 — 현재 코드는 이 리스트를 만든 뒤 바로 `Counter`로 넘기므로, 순서
리스트를 변수로 유지하도록 살짝 리팩터링 필요).

**알고리즘 (신규 함수, 가칭 `_extract_sequences`)**:
1. 세션의 정규화된 커맨드 리스트에서, 서로 다른 두 커맨드 A, B의 모든
   순서쌍(A, B)에 대해 "갭 허용 매칭 횟수"를 센다: 리스트를 한 번 순회하며
   포인터가 A를 가리키다가 A를 만나면 B를 가리키도록 전진, B를 만나면 매칭
   1회 카운트 후 포인터를 다시 A로 리셋(non-overlapping, greedy).
2. 매칭 횟수가 임계값(2) 이상인 (A, B) 쌍만 "2단계 후보"로 남긴다.
3. 각 2단계 후보 (A, B)에 대해, B 다음에 오는 C를 추가해 (A, B, C) 3단계로
   확장을 시도하고 같은 방식으로 매칭 횟수를 센다. 임계값을 넘으면 3단계
   후보로 채택하고 원래의 2단계 후보 (A, B)는 버린다(가장 긴 체인만 보고).
   같은 방식으로 4단계까지 반복.
4. 최종적으로 채택된 각 시퀀스를 `summary_lines`에
   `- "A" → "B" → "C" 순서로 N번 반복 실행함` 형태로 추가.

**복잡도**: 세션당 커맨드 종류가 적다는 전제(실제 세션 로그 특성상 distinct
커맨드 수는 보통 수십 개 이내) 하에 O(distinct²  × 세션 길이) 수준으로,
현재 세션 규모에서 문제없다.

**테스트**: 기존 `test_preprocessing.py`에 케이스 추가 — 갭이 있어도 매칭됨 /
2단계 임계값 미달 시 후보 안 됨 / 3단계로 확장 성공 시 2단계 버전은 결과에
안 나옴 / 4단계 초과는 확장 안 함.

### 2. AI서버 — 스키마 + 프롬프트 (`ai_server/schemas.py`, `ai_server/gemini_client.py`)

```python
class SkillRecommendation(BaseModel):
    type: Literal["skill"]
    skill_name: str              # kebab-case
    skill_description: str       # SKILL.md frontmatter용 한 줄 설명
    suggested_steps: str         # SKILL.md 본문 (마크다운)
    reason: str
    confidence: Literal["low", "medium", "high"]


class GeminiSkillCandidate(BaseModel):
    # GeminiHookCandidate/GeminiClaudeMdCandidate와 동일한 이유로 type 없음
    skill_name: str
    skill_description: str
    suggested_steps: str
    reason: str
    confidence: Literal["low", "medium", "high"]
```

`Candidate = Union[HookCandidate, ClaudeMdCandidate, SkillRecommendation]`로
확장. `GeminiAnalyzeSchema`에 `skill_candidates: list[GeminiSkillCandidate]`
추가, `_to_analyze_response`에 skill 리스트 조립 로직 추가(hook/claude_md와
동일 패턴 — 리스트 소속으로 `type` 채움).

`SYSTEM_INSTRUCTION`에 판단 기준 3번째 항목 추가:

> - skill: 2단계 이상 이어지고, 중간에 판단/분기가 섞이는 반복 절차. "이
>   순서로 이렇게 해야 한다"는 여러 문장으로 설명해야 하는 경우. 최소 2회
>   이상 반복된 것만 후보로 삼아라(hook/claude_md는 3회 기준이지만 skill은
>   2회).

**테스트**: `test_schemas.py`에 `SkillRecommendation`/`GeminiSkillCandidate`
스키마 테스트, `test_gemini_client.py`에 skill 후보 조립 테스트(hook/claude_md
혼합 응답에서도 정확히 분류되는지).

### 3. 데이터 모델 + 매칭 (`web-server/models.py`, `web-server/matching.py`)

**신규 테이블**:
```python
class Skill(SQLModel, table=True):
    __tablename__ = "skills"
    id: str = Field(default_factory=new_id, primary_key=True)
    project_id: str = Field(foreign_key="projects.id")
    name: str                      # kebab-case
    description: str               # SKILL.md frontmatter 설명
    steps_content: str             # SKILL.md 본문 마크다운
    created_at: datetime = Field(default_factory=datetime.utcnow)
    updated_at: datetime = Field(default_factory=datetime.utcnow)
```

**기존 테이블 확장**:
- `ProjectRevision.target`: `"content"|"hooks"` → `"content"|"hooks"|"skill"`.
  `skill_id: Optional[str] = Field(default=None, foreign_key="skills.id")` 추가
  (target="skill"일 때만 사용, content/hooks 리비전은 계속 `None`).
- `RecommendationGroup.type`에 `"skill"` 값 허용(이미 `str` 타입이라 스키마
  변경 불필요, 문서화만).

**매칭 (`match_skill_candidate`, 신규)**: `match_claude_md_candidate`와 거의
동일한 구조 — `ai_client.embed(skill_description)` 호출 → 기존 skill 타입
그룹들과 코사인 유사도 비교 → ≥0.85면 합류, 아니면 신규 그룹 생성.
`representative_text`=`skill_description`, `representative_vector`=그 임베딩.
2명 이상 합류 시 기존 `PROMOTION_THRESHOLD`로 그대로 `promoted=True`.

**DB 마이그레이션**: 새 컬럼/테이블 추가 — `SQLModel.metadata.create_all()`은
새 테이블(`skills`)은 만들어주지만 기존 `project_revisions`에 `target` 값
확장(문자열이라 스키마 변경 자체는 없음)과 `skill_id` 컬럼 추가는 로컬
`app.db`에 지난 두 번(온보딩 스프린트, hooks 분리 스프린트)과 동일하게
`ALTER TABLE`을 수동으로 한 번 더 실행해야 한다.

**테스트**: `test_matching.py`에 skill 매칭 테스트 세트(claude_md 테스트
세트와 동일 패턴 — 유사도 높으면 합류/낮으면 별도 그룹/2명 이상 승격).

### 4. 웹서버 API (`web-server/routers/`)

- **업로드 오케스트레이션**: `routers/sessions.py`의 `upload_session`에서
  후보 순회 시 `candidate["type"] == "skill"`이면 `match_skill_candidate`
  호출(hook/claude_md와 동일한 분기 구조에 한 줄 추가).
- **적용**: 기존 `apply_personal_recommendation`/`apply_recommendation_group`
  엔드포인트에 `type=="skill"` 분기 추가 — payload에서 `skill_name`/
  `skill_description`/`suggested_steps`를 꺼내 새 `Skill` row를 즉시 생성하고
  커밋(hook/claude_md처럼 "적용 후 별도 저장 필요"가 아니라, 스킬 자체가
  새로 생기는 것이므로 생성은 즉시 커밋 — 이후 수정은 아래 PUT으로).
- **신규 라우터** (`web-server/routers/skills.py`):
  - `GET /projects/{project_id}/skills` — 목록(멤버만).
  - `GET /projects/{project_id}/skills/{skill_id}` — 단건 조회.
  - `PUT /projects/{project_id}/skills/{skill_id}` — `name`/`description`/
    `steps_content` 수정 + `ProjectRevision(target="skill", skill_id=...)`
    생성(멤버만, `update_project_content`와 동일 패턴).
  - `DELETE /projects/{project_id}/skills/{skill_id}` — 삭제(멤버만). AI가
    잘못 만든 스킬을 정리할 수 있어야 하므로 포함.
- **리비전 필터 확장**: 기존 `GET .../revisions?target=`에 `skill_id` 쿼리
  파라미터 추가 — `target=skill`일 때 특정 스킬의 리비전만 보이도록.
- **PUSH 확장**: `push_to_github`에서 기존 CLAUDE.md + `.claude/settings.json`
  push 이후, 프로젝트의 모든 `Skill`을 순회하며
  `github_client.push_file(path=f".claude/skills/{skill.name}/SKILL.md", ...)`
  호출(N개 스킬 = N번 추가 호출). **알려진 한계(기록만, 이번 스코프 아님)**:
  스킬 이름을 바꾼 뒤 PUSH하면 레포에 예전 이름의 디렉토리가 남는다(삭제
  안 됨) — hooks 분리 스프린트의 "그룹 대표값 고정" TODO와 같은 성격의
  후속 과제로 남겨둠.

**테스트**: `test_skills.py`(신규) — 목록/조회/수정+리비전 생성/삭제/멤버
아니면 403. `test_sessions.py`에 skill 후보 적용 시 `Skill` row 생성 테스트
추가. `test_projects.py`(또는 `test_push.py`)에 PUSH 시 스킬 개수만큼
`push_file` 추가 호출 테스트.

### 5. 프론트엔드

- **`lib/projects.ts`**: `Skill` 타입, `listSkills`/`getSkill`/`saveSkill`/
  `deleteSkill` 함수, `SkillCandidatePayload` 타입(`skill_name`/
  `skill_description`/`suggested_steps`), `listRevisions`의 `target`에
  `"skill"` 추가 + `skillId?` 파라미터.
- **`app/project/[id]/page.tsx`**: 좌측 section에 3번째 탭 **"Skill"** 추가
  (기존 CLAUDE.md/Hooks 탭과 같은 스타일). 탭 선택 시:
  - 스킬 목록(이름 리스트, 클릭 시 선택)
  - 선택된 스킬의 편집기 — **name/description/steps_content 3개 필드**
    그대로 노출(이름 텍스트 입력, 설명 텍스트 입력, steps 마크다운
    textarea). 단일 SKILL.md 원문 textarea로 합치는 방식은 채택 안 함 —
    그러면 프론트-백엔드 사이에 frontmatter 파싱/직렬화 계층이 하나 더
    필요해지는데, PUT 요청 바디가 이미 3개 필드 그대로라 그럴 이유가 없음
    (Simplicity First)
  - "저장" 버튼 → `saveSkill`
  - "삭제" 버튼 → `deleteSkill`
  - 변경 이력 박스는 선택된 스킬 기준으로 필터링해서 표시
- **추천 카드**: `type==="skill"`인 개인/팀 추천 카드 렌더링 추가
  (`skill_name`/`skill_description`/`suggested_steps` 미리보기 + reason).
  "적용하기" 클릭 시 API 호출 → 성공하면 Skill 탭으로 자동 전환하고 방금
  생성된 스킬을 선택 상태로 보여줌(사용자가 바로 확인/수정할 수 있게).

## 스코프 밖 (이번 설계에서 명시적으로 제외)

- **tool_use 기반 시퀀스**(Edit/Write 등 bash 외 도구 조합) — v2. v1은 bash
  커맨드 시퀀스만.
- **수동 "새 스킬 만들기"** — v1은 AI 후보 적용으로만 생성. 생성 후 편집/삭제는
  지원.
- **PUSH 시 스킬 이름 변경에 따른 예전 경로 정리** — 알려진 한계로 기록,
  후속 과제.
- **skill 시퀀스 탐지 성능 최적화**(distinct 커맨드 수가 매우 많은 세션 대비) —
  현재 세션 규모에서 문제없다고 판단, 필요해지면 재검토.
- **그룹 대표값 재계산**(DESIGN.md 기존 TODO, hook/claude_md와 동일하게 skill도
  해당) — 이번 스프린트 스코프 아님, 기존 TODO 그대로 유지.
