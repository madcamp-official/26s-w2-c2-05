# Design: 온보딩(Stage 0) 추가 + 웹서버-AI서버 연동(Stage 1)

Generated by superpowers:brainstorming, 2026-07-13
Status: DRAFT (승인 대기)

## 배경 — 왜 이 문서가 필요한가

`ai_server`는 이미 완성돼 있다(`/analyze`, `/embed`, 24개 테스트 통과,
2026-07-13 스프린트 완료). 그런데 `web-server`를 실제로 열어보니 두 가지가
확인됐다:

1. **세션 업로드 → AI 분석 연동 자체가 아직 없다.** `web-server/main.py`엔
   `projects`/`auth`/`github_auth`/`presence` 라우터만 등록돼 있고, 세션
   업로드 엔드포인트나 `ai_client.py`가 없다.
2. **`DESIGN.md`에 적힌 팀 매칭 아키텍처(`RecommendationGroup`/
   `GroupMembership`/임계값 승격)가 실제로 구현된 모델과 다르다.**
   실제 `web-server/models.py`는 훨씬 단순하다:
   - `Project.content` — CLAUDE.md 텍스트 전체를 통째로 저장하는 필드
     하나 (공유 문서 방식, 알고리즘 매칭 없음)
   - `ProjectMember.role`("owner"/"member") — 이미 완전히 동작 중
     (생성자=owner, `POST /projects/{id}/invite`로 멤버 초대, rename/
     delete/set_github_repo는 owner 전용으로 이미 막혀 있음)
   - `ProjectRevision` — content 수정 이력만 기록

   즉 팀원은 "여러 명 세션을 매칭해서 임계값 넘으면 자동 승격"이라는
   복잡한 모델 대신, **"멤버 누구나 편집 가능한 공유 CLAUDE.md 문서 +
   owner만 GitHub에 push"**라는 훨씬 단순한 모델로 이미 방향을 잡아둔
   상태다. 이 문서는 이 **실제 모델을 기준으로** AI 연동을 설계한다.
   (`DESIGN.md`의 "DB 스키마" 절은 이 실제 구현과 어긋나 있어 별도로
   바로잡을 필요가 있음 — 이 문서의 스코프는 아니지만 기록해 둠.)

이와 별개로, 사용자가 유저플로우에 **Stage 0(온보딩)** 을 추가하고 싶어함
— 세션을 시작하기 전에 초기 CLAUDE.md 규칙을 먼저 세우고 싶다는 요구.

## Stage 0와 2026-07-11 기각 결정의 관계

`DESIGN.md`는 2026-07-11에 "프로젝트 생성 시 특성 선택 → 기본 md/hook
템플릿 자동 생성"을 다음 두 이유로 명시적으로 제거한 바 있다:
1. 타겟 유저(컨벤션 모르는 주니어/취미 개발자)에게 선택 부담을 지운다
2. 핵심 가치(세션 로그 기반 개인화 추천)와 반대 방향인 **정적** 템플릿

이번 Stage 0 설계는 아래 두 가지로 이 우려를 해소한다:
1. **선택 부담 완화**: 자유 서술형 질문이 아니라 "잘 알려진 개발
   원칙"을 체크리스트로 보여주고 고르게 한다 — 회상(recall)이 아니라
   인식(recognition) 기반이라 부담이 적다. (레포 자동 스캔은 검토했지만
   기각 — 세션 시작 전이면 레포에 스캔할 게 없을 수 있고, GitHub OAuth
   연결이나 새 로컬 스크립트라는 불필요한 선행 의존성이 생기기 때문)
2. **정적 템플릿 아님**: 선택 항목 + 자유 텍스트 요구사항을 Gemini
   호출 1회로 자연스럽게 다듬어서(우리 실제 `CLAUDE.md`를 만든 방식과
   동일한 패턴) 생성 — 입력이 다르면 결과물도 다르다.

## 아키텍처 개요

```
[프로젝트 생성 시, 세션 업로드 전 — Stage 0, 1회만]
   ↓
[온보딩 폼] 개발 원칙 체크리스트(다중 선택) + 팀/개인 + 탭/스페이스
           + "기타 요구사항" 자유 텍스트
   ↓
[AI서버: POST /generate-base-claude-md] (신규)
   선택 항목 + 자유 텍스트 → Gemini 호출 1회 → 자연스러운 CLAUDE.md
   텍스트 생성 (discriminated union 아님, 출력은 markdown 문자열 하나뿐
   — 우리가 T-08에서 겪은 anyOf/const 제약과 무관)
   ↓
[웹서버] project.content를 이 결과로 설정 (기존 DEFAULT_MD 대체)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[Stage 1 — 세션마다 반복]
   ↓
[프론트] 세션 JSONL 업로드
   ↓
[웹서버: POST /projects/{id}/members/{member_id}/sessions] (신규)
   전처리(규칙 기반) → ai_server의 /analyze 호출 → 후보(hook/claude_md)
   ↓
[프론트] 후보를 "제안 카드"로 표시 (자동 반영 아님)
   ↓
멤버가 claude_md 후보를 "적용" 클릭
   ↓
[웹서버: 기존 PUT /projects/{id}] content에 반영 (멤버 누구나 가능,
기존 권한 그대로)
   ↓
owner가 검토 후 "push" 클릭
   ↓
[웹서버: 기존 POST /projects/{id}/push] — **owner 전용으로 수정 필요**
(현재는 멤버 아무나 호출 가능한 버그, 아래 참고)
```

**hook/skill 후보**: 이번 스코프에서는 제안 카드로 **표시만** 하고
적용/저장/push는 하지 않는다. 이유: `Project`에 hook을 저장할 필드
(`hooks_json` 같은)가 없고, `push_to_github`도 CLAUDE.md 파일 하나만
쓰도록 짜여 있다. hook 저장/push는 다음 스프린트로 미룬다.

## 컴포넌트 상세

### 1. AI서버 — `POST /generate-base-claude-md` (신규)

**스키마** (`ai_server/schemas.py`에 추가):
```python
class OnboardingRequest(BaseModel):
    principles: list[str]          # 예: ["tdd", "conventional_commits", "code_review_required"]
    tech_stack: str                # 자유 텍스트, 예: "Python, FastAPI"
    team_or_individual: Literal["team", "individual"]
    indent_style: Literal["tabs", "spaces"]
    custom_requirements: str = ""  # 자유 텍스트, 없으면 빈 문자열


class OnboardingResponse(BaseModel):
    base_claude_md: str
```

**동작**: 기존 `call_gemini_analyze`와 같은 패턴(타임아웃 15초, 재시도
1회, `gemini_analyze_limiter`/`gemini_analyze_rpd_counter` 재사용 —
새 limiter 안 만듦, 같은 생성 모델 쿼터를 공유). `response_schema`는
`OnboardingResponse` 그대로 써도 된다 — 필드가 `base_claude_md: str`
하나뿐이라 Union도 단일값 Literal도 없어서 T-08에서 겪은 스키마 제약과
무관하다.

**시스템 프롬프트 원칙**: 선택된 원칙 + 자유 텍스트를 "왜 이 규칙을
골랐는지"까지 설명하지 않고, 우리 실제 `ai_server`에서 이미 검증한
패턴대로 담백하게 규칙 목록 형태로 정리 — 프롬프트당 요구사항 최소화
원칙(섹션5) 유지.

### 2. 웹서버 — 온보딩 트리거

`POST /projects` (기존) 흐름에 온보딩 단계를 어떻게 끼워 넣을지는 두
가지 방식이 가능하다:
- (A) 프로젝트 생성과 동시에 온보딩 폼도 같이 제출 → `create_project`
  핸들러가 AI서버를 호출해서 `content`를 채움
- (B) 프로젝트는 먼저 `DEFAULT_MD`로 생성되고, 온보딩은 별도
  `POST /projects/{id}/onboarding` 엔드포인트로 나중에 채움

**권장: (B)** — 기존 `create_project`(이미 안정적으로 동작 중, 여러
라우터가 의존)를 건드리지 않고 새 엔드포인트 하나만 추가하는 게
Surgical Changes에 맞다. 프론트에서는 프로젝트 생성 직후 바로 온보딩
폼으로 이동시키면 사용자 경험상 (A)와 차이 없음.

### 3. 웹서버 — 세션 업로드 파이프라인 (신규, 원래 플랜 Task 3/4/6 재사용 — Task 5 제외)

`docs/superpowers/plans/2026-07-10-web-server-frontend.md`의 Task 3
(전처리)·Task 4(`ai_client.py`)는 그대로 유효하다 — AI서버 계약과
무관한 순수 로직/HTTP 클라이언트라서 팀 매칭 모델 변경의 영향을 안
받는다. **Task 5(매칭/병합 로직)는 스킵** — 임계값 승격 개념 자체가
없어졌으므로.

Task 6(세션 업로드 오케스트레이션)은 아래처럼 단순화된다:
- `POST /projects/{project_id}/members/{member_id}/sessions` 업로드
- 전처리 → 패턴 없으면 `status: "no_patterns"`로 즉시 응답
- 패턴 있으면 `ai_client.analyze()` 호출 → 후보 리스트를 **그대로**
  응답에 실어 반환 (DB에 `RecommendationGroup` 매칭/저장하지 않음 —
  프론트가 그 응답을 받아서 "제안 카드"로 표시하고, 사용자가 적용을
  누르면 그때 기존 `PUT /projects/{id}`를 별도로 호출)
- `PersonalRecommendation` 테이블은 유지해도 되고(누가 어떤 제안을
  받았는지 이력), 안 만들어도 무방 — 이번 스코프에 필수는 아님

### 4. 권한 버그 수정 — `POST /projects/{id}/push`

현재 `web-server/routers/projects.py:276`:
```python
if db.get(ProjectMember, (project_id, user.user_id)) is None:
    raise HTTPException(status_code=403, detail="접근 권한이 없습니다")
```
멤버이기만 하면 누구나 push를 트리거할 수 있다 — "owner만 push 결정/
실행"이라는 이번 요구사항과 어긋난다. `member.role != "owner"` 체크로
바꿔야 한다 (`rename_project`/`delete_project`가 이미 쓰는 패턴과 동일).

## 스코프 밖 (이번 설계에서 명시적으로 제외)

- 레포 자동 스캔, GitHub OAuth를 Stage 0에 연동하는 것 — 온보딩은
  질문/선택만으로 완결
- hook/skill 후보의 저장·push (표시만 함, 다음 스프린트)
- 개인 모드/팀 모드 아키텍처 자체의 재검토 — 이번 변경과 별개 축
- `DESIGN.md`의 "DB 스키마"/"점진적 매칭" 절을 실제 코드에 맞게
  갱신하는 작업 — 이 문서에서 발견은 기록하되, 실제 정정은 별도 작업
