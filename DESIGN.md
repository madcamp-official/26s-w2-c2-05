## 추천 타입 3종 (2026-07-10 확정 — Skill 추천 타입 추가)

원래 피치는 "hook/CLAUDE.md/**skill** 추천"이라고 세 가지를 말했지만, 지금까지의
스키마 설계는 hook과 claude_md 두 가지만 다뤘다. 이유: 세션 로그에서 발견되는
반복 패턴 중 **여러 단계로 이어지고 중간에 판단이 섞이는 워크플로우**(예: "마이그
레이션 → 시드 갱신 → 서버 재시작을 항상 이 순서로")는 hook(단일 이벤트→단일 명령,
판단 없음)에도 claude_md(배경 지식 한 줄)에도 억지로 끼워맞춰야 했다. 이건 원래
Claude Skill이 담당하는 영역이므로, 세 번째 추천 타입 `SkillRecommendation`을
스키마에 추가한다.

```python
class SkillRecommendation(BaseModel):
    scope: Literal["personal", "team"] = "personal"
    affected_members: int | None = None
    type: Literal["skill"]
    reason: str                  # plain language 설명
    skill_name: str              # 제안 스킬 이름 (kebab-case)
    skill_description: str       # SKILL.md frontmatter용 한 줄 설명
    suggested_steps: str         # SKILL.md 본문에 들어갈 절차 (마크다운)
    confidence: Literal["low", "medium", "high"]
```

**판단 기준 (3분류로 확장)**:
- hook: 매번 결정론적으로 자동 실행돼야 하는 단일 트리거→단일 명령. 판단 불필요.
- claude_md: 세션마다 컨텍스트로 스며들면 되는 단일 사실/선호. 절차가 아님.
- **skill (신규)**: 2단계 이상 이어지고, 중간에 판단/분기가 섞이는 반복 절차.
  "이 순서로 이렇게 해야 한다"는 여러 문장으로 설명해야 하는 경우.

**스코프 영향(정직하게 기록)**: 프론트 카드 렌더링 분기가 하나 늘고, 유저별 추출
프롬프트의 few-shot 예시도 3분류로 늘어야 한다. HOLD SCOPE로 이미 확정했던 범위
위에 추가되는 작업량이라, 개인 모드의 hook/claude_md 두 타입을 먼저 안정화한
뒤에 skill 타입을 얹는 순서를 권장한다 (아래 Next Steps 참고).

## Problem Statement

비개발자 또는 그룹 프로젝트 팀원들은 Claude Code의 hook/CLAUDE.md/skill 같은
엔지니어링 컨벤션을 잘 모른다. 그 결과:
- AI를 비효율적으로 써서 토큰을 낭비하거나
- 팀원마다 제각각의 방식으로 AI를 써서 결과물 스타일이 통일되지 않는다.

## What Makes This Cool

세션 로그만 올리면 그 사람(또는 팀)에게 맞춘 hook/CLAUDE.md가 **자동으로 생성**된다는
것. 특히 팀 버전에서는, 여러 명이 각자 세션을 올리면 "이 팀은 이렇게 일하는구나"를
발견해서 팀 전체가 공유할 컨벤션을 뽑아준다 — 개인용 세션 분석기는 이미 여럿 있지만,
"여러 명의 세션을 병합해서 팀 공통 환경을 만들어주는" 도구는 없다.

## Premises

1. 타겟은 확정 — 엔지니어링 컨벤션을 모르는 비개발자/팀원. 번역 UX가 핵심 가치.
2. 차별화 포인트는 개인 추천이 아니라 **여러 명의 세션을 모아 팀 단위로 통일된
   개발 환경을 구축**하는 것.
3. 포트폴리오 어필 포인트: "우리가 Agent AI 워크플로우를 이해하고 있다"를
   보여주는 결과물이기도 함.
4. 서브 스토리: 토큰 낭비 방지 (초보자의 비효율적인 AI 사용 패턴을 잡아줌).

## Approaches Considered

### Approach A: 개인 코치 + 팀은 데모용 목업
개인 세션 업로드→추천 플로우만 탄탄히 만들고, 팀 화면은 준비된 데이터로 보여주기만.
공수 작음, 리스크 낮음. 단점: 차별화가 얕아 보임 — 심사위원이 파고들면 목업인 게 티남.

### Approach B: 진짜 팀 환경 빌더 (채택)
같은 프로젝트에 속한 여러 명이 각자 JSONL을 업로드 → 유저별 패턴 추출 →
"몇 명에게서 몇 번 반복됐는지" 기준으로 실제 병합 → 팀 공통 hook/CLAUDE.md 추천.
공수 중간~큼, 리스크 중간(다중 세션 테스트 데이터 확보, 병합 임계값 튜닝).
장점: 기존 도구에 없는 각도, 실제 팀 페인포인트(온보딩/컨벤션 통일)를 건드림.

### Approach C: 개인용 + 즉시 적용 UX 극한 추구
팀 기능은 로드맵으로 미루고 "적용" 버튼으로 실제 파일이 생성되는 경험에 집중.
공수 중간, 리스크 낮음. 단점: 이미 나온 가장 뾰족한 차별화 스토리를 포기하는 셈.

## Recommended Approach: B — 팀 환경 빌더

### 병합(merge) 로직 — 배치가 아니라 점진적(incremental) 매칭 (팀원 워크플로우 반영, 2026-07-10)

기존 스키마의 `scope`/`affected_members` 필드에 실제 값을 채우는 문제로 축소된다.
단, **"팀원 전원이 업로드할 때까지 기다렸다가 한 번에 병합"하는 배치 방식이 아니라,
세션이 하나 들어올 때마다 그 즉시 처리하는 점진적 방식**이다 — 팀원들이 동시에
업로드할 이유가 없으니 이 구조가 자연스럽고, 매번 전체를 다시 계산할 필요도 없다.

1. **추출**: 새 세션 업로드 → 전처리(규칙 기반) → Gemini `flash-lite` 구조화 생성
   호출 1회 → 후보(`suggested_text` / hook 커맨드) 추출. 개인 모드와 완전히 동일한
   경로 — 팀 모드는 이 위에 매칭 단계만 얹는다.
2. **매칭**: 추출된 각 후보를 **기존 그룹들의 대표 임베딩과 비교**한다.
   - Hook 후보: 정규화한 `command` 문자열이 기존 그룹과 일치하면 그 그룹에 합류.
   - CLAUDE.md 후보: Gemini 임베딩 모델(`task: clustering`)로 벡터화 후 기존 그룹
     대표 벡터와 코사인 유사도 비교. 유사한 그룹이 있으면 합류(멤버로 등록,
     멤버 식별은 `user_id`), 없으면 새 그룹 생성(멤버 1명).
   - **병합용 재작성 LLM 호출은 없음** — 그룹 대표 문구는 그룹 내 원문을 그대로 사용.
3. **임계값 체크**: 방금 갱신된 그룹의 멤버 수만 확인 (전체 재계산 안 함) — 기준
   (기본 2명 이상) 넘으면 승격. **바뀐 그룹만 부분 렌더링.**
4. **반영 — 자동 반영 + 근거 표시, 메커니즘 확정 (2026-07-10, /plan-eng-review에서 구체화)**:
   임계값 넘은 그룹은 유저 승인 버튼 없이 **웹 UI 상의 병합된 CLAUDE.md/hooks 상태가
   즉시 갱신**된다 (+ "왜 반영됐는지" 근거 표시: 몇 명의 세션에서 나온 패턴인지,
   Gemini가 생성한 reason). **단, 웹앱은 유저 로컬 파일시스템에 직접 쓸 수 없으므로,
   실제 로컬 반영은 유저가 다운로드/복사해서 직접 저장하는 수동 단계다.** "자동"은
   "웹 UI 상태가 승인 버튼 없이 자동 갱신된다"는 뜻이지 "로컬 파일이 자동으로
   바뀐다"는 뜻이 아니다. GitHub API로 팀 저장소에 직접 커밋하는 진짜 자동 반영은
   **v2 로드맵으로 이관** — OAuth/토큰 관리라는 추가 인프라가 1주일 예산엔 안 맞음.
   임계값 못 넘은 그룹은 로그에 남아있다가, 나중에 멤버가 더 붙으면 그때 승격된다.
5. **충돌 처리**: 정교한 다수결/최신우선 로직은 이번 캠프 기간엔 넣지 않는다 —
   그룹 안에서 가장 많이 반복된 원문을 대표로 쓰는 것으로 충분.

### 프롬프트 설계 원칙 (팀원 관찰, 채택)

Gemini flash-lite는 한 프롬프트에 요구사항이 많아질수록 각 요구사항의 결과 품질이
떨어지는 경향이 있다. 그래서 이미 결정한 "병합용 재작성 호출을 아예 없앤 것"과
같은 방향으로, **유저별 추출 호출 자체도 한 번에 시키는 일의 개수를 최소화**하는
걸 원칙으로 삼는다 (예: 분류+이유 설명+커맨드 생성을 억지로 다 우겨넣지 말고,
스키마와 few-shot으로 범위를 좁게 유지). 구체적인 프롬프트 분리 여부는
`/plan-eng-review`에서 확정.

### Gemini 모델/무료 티어 제약

**모델: `gemini-2.5-flash-lite`로 전환 (2026-07-10 확정, 팀원 안 채택)** — 원래
스펙은 `gemini-2.5-flash`였으나 비용/속도를 더 아끼기 위해 flash-lite로 변경.
**주의(리스크로 기록)**: flash-lite는 flash보다 판단력·자연어 설명 품질이 떨어질
수 있음 — hook vs claude_md 분류나 reason 문장 품질이 낮아지면 데모 중 눈에 띌 수
있으니, 초기 구현 후 실제 출력 품질을 꼭 확인하고 필요하면 flash로 되돌릴 것.

공식 문서(ai.google.dev)는 계정별 실시간 한도를 AI Studio 대시보드에서만 보여주고
고정 표를 공개하지 않는다. 서드파티 소스들은 flash 계열 기준 10~15 RPM,
250~1500 RPD 사이로 엇갈리게 보고한다 — **정확한 값은 팀 계정의 AI Studio
대시보드에서 직접 확인할 것.** 점진적 매칭 구조상 한 번에 여러 호출이 몰릴 일이
애초에 적고(세션 하나 들어올 때마다 호출 1회), 병합 단계는 임베딩(별도 쿼터)만
쓰므로 생성 모델 RPM 예산을 건드리지 않는다.

### 처리 방식 확정 (2026-07-10, /plan-eng-review)

업로드 처리는 **동기**로 간다 — 백그라운드 큐/워커 인프라는 이 프로젝트 규모
(수업 팀 프로젝트, 팀원 소수, 동시 업로드가 몰릴 일이 거의 없음)에서는 과한
복잡도라고 판단. 대신:
- Gemini 호출에 타임아웃(15초)을 걸어 무한 대기를 방지 — 넘으면 D5에서 결정한
  "1회 재시도 → 실패 메시지" 경로로 처리.
- Gemini 호출 함수 앞에 `aiolimiter`(`AsyncLimiter`) 라이브러리로 RPM 보호 —
  최근 60초 안의 호출 수를 세서 한도(정확한 값은 AI Studio에서 확인 후 여유
  있게 설정, 예: 8~10) 넘으면 자동으로 잠깐 대기 후 호출. 서버가 KAIST VM
  한 대에서만 도는 단일 프로세스라 Redis 등 분산 속도 제한 인프라는 불필요.

### 왜 A/C보다 나은가

A는 원래 스펙에서 이미 "로드맵"으로 분류했던 걸 데모용 무대장치로만 쓰는 셈이라
얕아 보일 위험이 있다. C는 좋은 UX지만 이번 대화에서 나온 가장 뾰족한 차별화
스토리(팀 통일 환경)를 스스로 포기하는 선택이다.

## Success Criteria

- 3명 이상의 (실제 또는 합성) 세션 로그를 업로드했을 때, 팀 공통 추천이
  "몇 명 중 몇 명에게서 나왔는지" 근거와 함께 표시된다.
- 개인 모드와 팀 모드가 같은 백엔드 파이프라인(전처리 → Gemini 구조화 분석)을
  공유하고, 팀 모드는 그 위에 병합 단계만 추가된 구조다.
- 비개발자가 결과 화면만 보고 무슨 뜻인지 이해할 수 있다 (hook/CLAUDE.md 용어를
  풀어서 설명).

## Next Steps (1주 기준 대략적 순서)

1. 개인 모드 파이프라인 먼저 완성 — hook/claude_md 두 타입만으로 전처리 →
   Gemini 구조화 분석 → 추천 표시. 이게 팀 모드의 추출 단계이기도 하므로
   선행 투자가 낭비되지 않는다.
2. 점진적 매칭 로직 구현 (새 세션 → 기존 그룹과 비교 → 합류/신규 생성 →
   임계값 체크 → 넘으면 자동 반영+근거 표시). 팀 모드용 합성 테스트 데이터로 검증.
3. 팀 결과 화면 — 승격된 그룹은 "N명이 반영했어요 + 근거", 미승격 그룹은
   로그로만 노출.
4. **SkillRecommendation 타입 추가** — hook/claude_md 두 타입이 안정화된 뒤에
   세 번째 타입(스키마, 프롬프트 few-shot, 프론트 카드)을 얹는다. 먼저 두 타입을
   흔들리지 않게 만들고 나서 확장하는 순서 — 처음부터 3분류로 시작하면 프롬프트
   품질(특히 flash-lite)이 애매한 케이스에서 흔들릴 위험이 있다.
5. **(시간 남으면 필수)** 프로젝트 생성 시 특성 선택 → 기본 md/hook 템플릿
   자동 생성 (팀원 안의 0단계). 데모 시작점을 빈 화면이 아니게 만드는 기능이라
   우선순위 높은 스트레치 목표.
6. (더 시간 남으면) Approach C의 "즉시 적용" UX를 개인 모드에도 추가.

### DB 스키마

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY, name TEXT, share_code TEXT UNIQUE, created_at TIMESTAMP
);
CREATE TABLE members (
    id TEXT PRIMARY KEY,              -- localStorage에 저장되는 UUID
    project_id TEXT REFERENCES projects(id),
    display_name TEXT, joined_at TIMESTAMP
);
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    project_id TEXT REFERENCES projects(id),
    member_id TEXT REFERENCES members(id),
    uploaded_at TIMESTAMP, status TEXT,   -- 'processed' | 'no_patterns' | 'failed'
    UNIQUE(project_id, member_id)         -- upsert 키: 최신 세션으로 교체 (D7)
);
CREATE TABLE recommendation_groups (      -- 팀 모드 병합 그룹
    id TEXT PRIMARY KEY,
    project_id TEXT REFERENCES projects(id),
    type TEXT,                            -- 'hook' | 'claude_md' | 'skill'
    representative_text TEXT,
    representative_vector TEXT,           -- JSON 배열 (claude_md/skill만)
    promoted BOOLEAN DEFAULT FALSE,        -- 임계값(2명) 넘었는지
    created_at TIMESTAMP, updated_at TIMESTAMP
);
CREATE TABLE group_memberships (          -- 그룹 소속 + 근거 표시용 원문
    group_id TEXT REFERENCES recommendation_groups(id),
    member_id TEXT REFERENCES members(id),
    session_id TEXT REFERENCES sessions(id),
    original_text TEXT, reason TEXT, confidence TEXT,
    PRIMARY KEY (group_id, member_id)
);
CREATE TABLE personal_recommendations (   -- 개인 모드 전용
    id TEXT PRIMARY KEY,
    session_id TEXT REFERENCES sessions(id),
    member_id TEXT REFERENCES members(id),
    type TEXT, payload TEXT,              -- JSON (hook/claude_md/skill 스키마 직렬화)
    created_at TIMESTAMP
);
```

### API 엔드포인트

| Method | Endpoint | 설명 |

### 배포 구조 (via /plan-eng-review, 2026-07-10)

웹 서버와 AI 서버를 분리하고, KCLOUD VM + Tunnel 환경(VPN 없이 외부 접속 가능해야
하므로)에 맞춰 설계. VM 위에서 프로세스 3개가 돈다:

```
                          Internet (심사위원 등)
                                 │
                    Cloudflare Tunnel — 라우트 1개만
                                 │
                                 ▼
                     [Next.js 서버 :3000]  ← 유일하게 외부 노출됨
                     (화면 서빙 + next.config.js
                      rewrites()로 /api/* 를 백엔드로
                      통째로 프록시. 코드 작성 없음)
                                 │  localhost 내부 호출
                                 ▼
                     [FastAPI 웹 서버 :8000]
                     (SQLite, API 7종, 비즈니스 로직)
                                 │  localhost 내부 호출만
                                 ▼
                     [FastAPI AI 서버 :8001]
                     (Gemini 생성/임베딩, aiolimiter)
                     터널 노출 없음 — 외부에서 절대 접근 불가
```

**결정 근거**: DB 붙은 API 서버를 인터넷에 직접 노출하지 않는 게 원칙(BFF/API
게이트웨이 패턴) — 진입점이 하나면 어뷰징 방지(우리 무료 티어 Gemini 쿼터를
관계없는 사람이 소진시키는 것 포함)를 한 곳에서만 신경 쓰면 됨. 부수 효과로
CORS 설정도 통째로 불필요해짐(브라우저는 항상 같은 origin에만 요청). `rewrites()`
설정 한 줄로 프록시가 끝나서 작업량도 추가되지 않음 — "더 탄탄한데 더 쉽기도 한"
드문 경우.

### 2인 팀 병렬 작업 전략 (서버 분리 반영, 2026-07-10)

프로세스가 3개(Next.js / 웹서버 / AI서버)로 나뉘면서 병렬화 축도 더 명확해졌다.
**AI 서버는 딱 2개 엔드포인트(`/analyze`, `/embed`)짜리 얇은 서비스라 가장
독립적으로 떼어낼 수 있는 조각** — DB를 전혀 모르고 입출력이 순수 함수에 가깝다.

- **한 명 = AI 서버**: Gemini/임베딩/`aiolimiter` 만 신경 쓰면 됨. 가짜 입력으로
  독립 테스트 가능, 웹서버 완성을 기다릴 필요 없음.
- **다른 한 명 = 웹서버 + Next.js**: 이 둘은 애초에 API 계약으로 강하게 묶여있어
  같은 사람이 맡는 게 자연스러움. 초반엔 AI 서버 대신 고정된 가짜 응답(mock)을
  반환하는 스텁을 넣어두고 진행하다가, AI 서버가 준비되면 실제 호출로 교체.

**먼저 고정해야 할 계약은 두 개**: ①위 API 엔드포인트 표(웹서버-프론트 계약),
②AI 서버의 `/analyze`, `/embed` 요청/응답 스키마(AI서버-웹서버 계약). 이 둘만
먼저 합의하면 이후로는 거의 충돌 없이 병렬 진행 가능.
